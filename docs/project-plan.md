## End state

A user opens `https://<username>.github.io/<repo-name>/` in any browser, anywhere, and sees a current Blue Jays 2026 dashboard. Data refreshes daily without their laptop being on. Anyone can fork the repo, edit `config.json`, point at their team, and run the same architecture for the team of their choice.

The token bill for the maintainer drops by ~95% versus the prior model. The dashboard's analyst voice is preserved — and is now the only thing Claude is ever asked to write.

## Architecture (locked)

Three runtime pieces, four files of state.

**Runtime:**

- `fetch_data.py` — pure fetcher. Reads `config.json`, calls MLB Stats API via the `MLB-StatsAPI` package, writes `data.json`. Has no opinions about who runs it.
- Wrapper scripts in `scripts/`:
  - `update_and_push.sh` — runs the fetcher, commits + pushes if `data.json` changed. Used by both the routine and (later) launchd.
  - `fetch_only.sh` — runs the fetcher, no git side effects. For manual / development runs.
- `index.html` — pure render layer. On load, fetches `data.json`, `notes.json`, and `config.json` (all same-origin, no CORS issues). Merges and renders. No external API calls from the browser.

**State files:**

- `config.json` — team identity. Team ID, league ID, division ID, primary color, dashboard title. The single file a forker edits.
- `data.json` — fact data, machine-generated by `fetch_data.py`. Standings, schedule, recent games, roster, player stats, transactions. Never hand-edited.
- `notes.json` — analyst voice, hand-written. Keyed by `gamePk`. Each entry has `moment` (one-line) and `meaning` (the paragraph). Empty entries are fine; dashboard renders facts-only when no note exists.
- `mlb-statsapi-reference.md` — internal reference (already written).

**Scheduler (Claude Code Routine):**

- Trigger: daily, 12:00 UTC (8:00 AM ET — after prior night's boxscores have settled).
- Action: prompt is "run `bash scripts/update_and_push.sh` and report any errors in one sentence; otherwise just say 'done'."
- Falls back cleanly to macOS `launchd` running the same `update_and_push.sh` if the user ever wants to leave the routine system.

### Phase 2 — Fetcher script + data schema

**Goal:** A working `fetch_data.py` that produces a valid `data.json` matching the schema the dashboard will expect.

**Steps:**

1. Create `config.json` at repo root with Jays values:

```json
{
  "team_id": 141,
  "team_name": "Toronto Blue Jays",
  "team_abbrev": "TOR",
  "league_id": 103,
  "division_id": 201,
  "season": 2026,
  "primary_color": "#134A8E",
  "accent_color": "#1d8acd",
  "dashboard_title": "Blue Jays 2026 Tracker",
  "brand_mark": "J"
}
```

2. Create `requirements.txt`:

```
MLB-StatsAPI>=1.9.0
```

3. Write `fetch_data.py`:
   - Reads `config.json`.
   - **Rule:** every team-specific value used in any API call comes from `config.json`. No hardcoded team IDs, league IDs, division IDs, or season values anywhere in this script. A forker editing `config.json` to switch teams must succeed without touching Python.
   - Uses `statsapi.get(endpoint, params)` (not the high-level helpers, which return display strings).
   - Pulls: division standings, wild card standings, schedule (past 30 + future 7), per-game boxscore for last 10 games, active roster + injury report, season stats for each active player, last 7 days of transactions.
   - Transforms each response into the shape `data.json` requires (defined below).
   - **Quality invariants** — before writing, assert:
     - Division standings has exactly 5 teams (every MLB division has been 5 teams since the 2013 realignment).
     - Wild card standings has ≥10 entries.
     - `recent_games` has ≥1 entry (unless season hasn't started — guarded by date check).
     - `team.record.w` + `team.record.l` equals games played as derived from the schedule.
     - No `None` / `NaN` in any required numeric field.
     - If any invariant fails, log the failure and exit non-zero. The wrapper sees the non-zero exit and skips the commit. Yesterday's `data.json` stays in place — better stale than wrong.
   - Writes to `data.json` atomically — write to `data.json.tmp`, then `os.replace` to `data.json`. Prevents a half-written file from breaking the dashboard if the script crashes mid-write.
   - Logs to stderr; non-zero exit if any required fetch fails or any invariant fails.

4. Define `data.json` schema. The exact shape Code should produce:

```
{
  "as_of": "ISO timestamp of fetch",
  "team": { record, place, last10, streak, runs_scored, runs_allowed, run_diff, pythag_w, pythag_l },
  "division": [ { team, w, l, pct, gb, streak, last10 }, ... ],
  "wild_card": [ { team, w, l, gb, note }, ... ],
  "recent_games": [
    { game_pk, date, home, opp, result, score, status, winning_pitcher, losing_pitcher, summary_facts }
  ],
  "upcoming_games": [
    { game_pk, date, home, opp, probable_pitcher_us, probable_pitcher_them }
  ],
  "roster": {
    "hitters": [ { id, name, pos, ab, avg, obp, slg, ops, hr, rbi, sb } ],
    "pitchers": [ { id, name, role, ip, era, whip, k, bb, w, l, sv } ]
  },
  "injuries": [ { name, status, eta_note } ],
  "transactions": [ { date, type, description, person_name } ],
  "run_diff_last_10": [ { date, diff, result } ]
}
```

   Code can extend this if a useful field is cheap; not contract-breaking until the dashboard reads it.

5. Pythagorean expectation is computed in Python: `pythag_pct = RS² / (RS² + RA²)`, then `pythag_w = round(pythag_pct * games_played)`. Not from API.
6. Numeric strings (`era`, `avg`, `ip` like `"6.2"`) are converted: floats for `era`/`avg`/`whip`; the `inningsPitched` string is parsed into decimal innings (`"6.2"` → `6.667`) only if used in math; for display, keep the string.

**Verification (manual):**

- Run `python fetch_data.py` locally.
- `data.json` exists, is valid JSON, has all sections.
- Spot-check: division standings include 5 AL East teams, Jays record matches reality, recent games include yesterday's game.
- Re-run; output is deterministic (same input data → same output, no random ordering).

**Handoff to Code:** Phase 3 starts when `data.json` is correct.

### Phase 3 — Dashboard refactor

**Goal:** `index.html` reads `data.json`, `notes.json`, and `config.json` on load. Inline `DATA` const is gone. UI is visually unchanged.

**Steps:**

1. Create `notes.json` with empty object `{}`. Don't try to backfill historical narratives — they were specific to a stale snapshot and would be more misleading than useful. Notes start fresh.
2. In `index.html`:
   - Delete the inline `DATA` const entirely.
   - Add `async function loadAll()` that fetches the three JSON files in parallel and stores results in a single module-level object.
   - **Schema validation:** at the top of `loadAll()`, after parsing `data.json`, assert that every expected top-level key exists (`as_of`, `team`, `division`, `wild_card`, `recent_games`, `upcoming_games`, `roster`, `injuries`, `transactions`, `run_diff_last_10`). If any is missing, `console.warn` loudly with the missing keys and render an inline error banner in the header. Catches fetcher/dashboard schema drift early.
   - Replace every `DATA.foo` reference with the equivalent from the loaded object.
   - For game rendering: merge `data.recent_games[i]` with `notes[game_pk]` if present. If no note, render the boxscore-derived facts only.
   - Wire `config.json` values to CSS variables for theming (`--primary-color`, `--accent-color`, `--brand-mark`).
3. Update the page title and brand from `config.dashboard_title` and `config.brand_mark`.
4. Add a small "Last refreshed: <as_of>" line in the header from `data.as_of` — replaces the deleted prose paragraph. **Staleness indicator:** if `as_of` is more than 36 hours old, render the line in amber. More than 72 hours, red. CSS-only; signals a broken pipeline that would otherwise decay silently.
5. Replace the old "lastRefreshed" prose with a structured "Since yesterday" panel:
   - Latest transaction(s) from `data.transactions`.
   - Last game's score and one-line summary.
   - Tonight's probable pitcher matchup from `data.upcoming_games[0]`.
6. Cache `data.json` to `localStorage` keyed by `as_of` so reopens within the same day are instant. Re-fetch if `as_of` is older than today.
7. **"Ask the analyst" chips — feature replacement.** The current artifact's chips call `window.cowork.askClaude`, which doesn't exist on GitHub Pages. Default treatment for v1 (recommended):
   - Keep the chips visually as they are.
   - On click: copy the question text to clipboard and show a toast — "Copied. Paste into Claude or your AI of choice."
   - Preserves the affordance, honest about what a static site can do, zero new dependencies.
   - **DECISION-OVERRIDE:** if you'd rather strip the chips entirely or wire a direct Anthropic API call (config-supplied key in `localStorage`), say so now. A direct-API version is ~50 lines and is a clean v2 follow-up; not blocking.

## Cross-cutting concerns

**Timezone:** routine runs at 12:00 UTC = 8:00 AM ET. Boxscores from games ending after midnight ET will be in by then. Pacific time games rarely run past 1 AM ET. If a game runs into extras and isn't settled by 8 AM ET, the next day's run picks it up.

**Doubleheaders:** the fetcher should treat both games of a doubleheader as separate entries with separate `gamePk`s. The schedule endpoint returns them as two games on the same date.

**Postponements:** games with `status.detailedState == "Postponed"` should be flagged in `data.json` so the dashboard can render them differently rather than treating an unplayed game as a result.

**Off-days:** when there's no game, `data.recent_games` doesn't grow but other fields update (standings, transactions). The routine should still commit if anything changed (standings always do, even on a Jays off-day, because other AL East teams played).

**Stat-school content:** the static reference material (pitch types, scoring codes, position diamond, stat definitions) stays inline in `index.html`. It's not data; it doesn't change. Moving it out adds complexity without benefit.

**Theming for forks:** primary color, accent color, brand mark character all live in `config.json`. Code applies them via CSS variables. Forker for another team edits one file.

- Player photos. Reachable from MLB.com but bandwidth-heavy; skip for v1.
- A mobile-specific layout. Current responsive CSS is good enough.
